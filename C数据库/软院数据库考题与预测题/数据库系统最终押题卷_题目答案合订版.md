## 一、基本概念题（20 分）

### 1. 填空题

**题目：**

1. 数据模型三要素是 ________、________ 和 ________。
2. 关系模型的三类完整性约束是实体完整性、________ 和 ________。
3. 关系代数中，________ 运算用于选出满足条件的元组，________ 运算用于选出指定属性列。
4. 事务 ACID 特性分别是原子性、________、隔离性和 ________。
5. 两段锁协议能够保证并发调度的 ________ 可串行化，但可能产生 ________。
6. 数据库恢复时，已提交事务需要 ________，未提交事务需要 ________。
7. Neo4j 常用的图查询语言是 ________。
8. MongoDB 中的 ________ 对应关系数据库中的表。
9. 分解到 ________ 范式通常可消除大部分插入、删除和修改异常。
10. BCNF 分解的主要代价是可能不保持 ________。

**答案：**

1. 数据结构；数据操作；完整性约束。
2. 参照完整性；用户自定义完整性。
3. 选择；投影。
4. 一致性；持久性。
5. 冲突；死锁。
6. REDO；UNDO。
7. Cypher。
8. 集合。
9. 第三范式 / 3NF。
10. 函数依赖。

### 2. 简答题

**题目：** 简述视图的概念、两个优点和两个局限。

**答案：**

视图是从一个或多个基本表或其他视图导出的虚表，其内容由查询定义。优点：可以简化复杂查询；可以隐藏敏感字段，提高安全性；可以提高数据逻辑独立性。局限：复杂视图可能影响查询性能；包含连接、聚合、分组等操作的视图通常不易更新。

## 二、数据查询题（40 分）

设有关系模式：

```text
Department(dNo, dName, addr, tel)
Student(sNo, sName, sex, age, dNo)
Course(cNo, cName, type, credit, dNo)
CourseRel(cNo, cPNo)
SC(sNo, cNo, score)
```

含义：`Department` 为学院，`Student` 为学生，`Course` 为课程，`CourseRel` 为课程和直接先修课程关系，`SC` 为学生选课成绩。

### 1. 关系代数

#### （1）查询“软件学院”所有学生的学号和姓名。

**答案：**

```text
π_sNo,sName(
  σ_dName='软件学院'
  (Department ⋈ Department.dNo=Student.dNo Student)
)
```

#### （2）查询选修了“数据库系统”课程的学生姓名和成绩。

**答案：**

```text
π_sName,score(
  σ_cName='数据库系统'
  (Student ⋈ Student.sNo=SC.sNo SC ⋈ SC.cNo=Course.cNo Course)
)
```

#### （3）查询选修了“数据库系统”但没有选修“编译原理”的学生学号。

**答案：**

```text
T1 = π_sNo(σ_cName='数据库系统'(SC ⋈ SC.cNo=Course.cNo Course))
T2 = π_sNo(σ_cName='编译原理'(SC ⋈ SC.cNo=Course.cNo Course))
T1 - T2
```

#### （4）查询选修了全部“必修”课程的学生学号和姓名。

**答案：**

```text
T1 = π_cNo(σ_type='必修'(Course))
T2 = π_sNo,cNo(SC)
T3 = T2 ÷ T1
π_sNo,sName(Student ⋈ Student.sNo=T3.sNo T3)
```

### 2. SQL

#### （5）给 `CourseRel` 补充主码和外码。

**答案：**

```sql
ALTER TABLE CourseRel
ADD CONSTRAINT pk_CourseRel PRIMARY KEY (cNo, cPNo);

ALTER TABLE CourseRel
ADD CONSTRAINT fk_CourseRel_course
FOREIGN KEY (cNo) REFERENCES Course(cNo);

ALTER TABLE CourseRel
ADD CONSTRAINT fk_CourseRel_prereq
FOREIGN KEY (cPNo) REFERENCES Course(cNo);
```

#### （6）创建视图：统计每门课程的课程号、课程名、选课人数、平均成绩。

**答案：**

```sql
CREATE VIEW v_course_score_stat AS
SELECT
    c.cNo,
    c.cName,
    COUNT(sc.sNo) AS studentCount,
    AVG(sc.score) AS avgScore
FROM Course c
JOIN SC sc ON c.cNo = sc.cNo
GROUP BY c.cNo, c.cName;
```

#### （7）将“软件学院”所有学生的“数据库系统”课程成绩提高 5 分，最高不超过 100 分。

**答案：**

```sql
UPDATE SC
SET score = CASE
    WHEN score + 5 > 100 THEN 100
    ELSE score + 5
END
WHERE sNo IN (
    SELECT s.sNo
    FROM Student s
    JOIN Department d ON s.dNo = d.dNo
    WHERE d.dName = '软件学院'
)
AND cNo IN (
    SELECT cNo
    FROM Course
    WHERE cName = '数据库系统'
);
```

#### （8）查询平均成绩大于等于 80 分的学生学号、姓名和平均成绩。

**答案：**

```sql
SELECT
    s.sNo,
    s.sName,
    AVG(sc.score) AS avgScore
FROM Student s
JOIN SC sc ON s.sNo = sc.sNo
WHERE sc.score IS NOT NULL
GROUP BY s.sNo, s.sName
HAVING AVG(sc.score) >= 80;
```

#### （9）查询“数据库系统”的直接先修课程名称。

**答案：**

```sql
SELECT p.cName
FROM Course c
JOIN CourseRel r ON c.cNo = r.cNo
JOIN Course p ON r.cPNo = p.cNo
WHERE c.cName = '数据库系统';
```

#### （10）查询每个学院的学生人数，输出学院名称和学生人数。

**答案：**

```sql
SELECT
    d.dName,
    COUNT(s.sNo) AS studentCount
FROM Department d
LEFT JOIN Student s ON d.dNo = s.dNo
GROUP BY d.dNo, d.dName;
```

## 三、数据库设计题（20 分）

**题目：**

某医院门诊管理系统需要保存如下信息：

1. 科室信息包括科室编号、科室名称、位置、电话。
2. 医生信息包括医生编号、姓名、职称。一个医生只属于一个科室，一个科室有多名医生。
3. 患者信息包括患者编号、姓名、性别、年龄、联系电话。
4. 患者可以预约医生门诊，需要记录预约时间、预约状态。
5. 患者就诊后形成就诊记录，需要记录就诊时间、诊断结果、医嘱。
6. 医生可以给患者开药，一次就诊可以开多种药品，需要记录药品数量和用法。
7. 药品信息包括药品编号、药品名称、规格、单价。

要求：

（1）写出 ER 模型中的实体、主要属性、联系和基数。

（2）转换为关系模式，标明主码和外码。

**答案：**

实体：

- 科室：科室编号、科室名称、位置、电话。
- 医生：医生编号、姓名、职称。
- 患者：患者编号、姓名、性别、年龄、联系电话。
- 预约：预约编号、预约时间、预约状态。
- 就诊记录：就诊编号、就诊时间、诊断结果、医嘱。
- 药品：药品编号、药品名称、规格、单价。

联系：

- 科室与医生：1:N。一个科室有多名医生，一个医生只属于一个科室。
- 患者与医生通过预约形成 M:N 联系，预约本身可作为联系实体。
- 患者与医生通过就诊记录形成 M:N 联系，一次就诊对应一个患者和一个医生。
- 就诊记录与药品是 M:N，一次就诊可开多种药品，一种药品可出现在多次就诊中，联系属性有数量和用法。

关系模式：

```text
Department(deptNo, deptName, location, tel)
```

主码：`deptNo`。

```text
Doctor(doctorNo, doctorName, title, deptNo)
```

主码：`doctorNo`。外码：`deptNo` 参照 `Department(deptNo)`。

```text
Patient(patientNo, patientName, sex, age, phone)
```

主码：`patientNo`。

```text
Appointment(appNo, patientNo, doctorNo, appTime, appStatus)
```

主码：`appNo`。外码：`patientNo` 参照 `Patient(patientNo)`，`doctorNo` 参照 `Doctor(doctorNo)`。

```text
Visit(visitNo, patientNo, doctorNo, visitTime, diagnosis, advice)
```

主码：`visitNo`。外码：`patientNo` 参照 `Patient(patientNo)`，`doctorNo` 参照 `Doctor(doctorNo)`。

```text
Medicine(medNo, medName, spec, price)
```

主码：`medNo`。

```text
Prescription(visitNo, medNo, quantity, usage)
```

主码：`(visitNo, medNo)`。外码：`visitNo` 参照 `Visit(visitNo)`，`medNo` 参照 `Medicine(medNo)`。

## 四、关系模式规范化题（10 分）

**题目：**

设关系模式：

```text
R(A, B, C, D, E, F)
```

函数依赖集：

```text
F = { A -> B, C -> D, D -> E, AC -> F }
```

求候选码，并将 R 分解为满足 BCNF 的关系模式。

**答案：**

先求候选码。

右部出现的属性有：

```text
B, D, E, F
```

没有出现在右部的属性是：

```text
A, C
```

所以候选码必须包含 `A` 和 `C`。

计算：

```text
(AC)+ = AC
由 A -> B 得 ABC
由 C -> D 得 ABCD
由 D -> E 得 ABCDE
由 AC -> F 得 ABCDEF
```

所以：

```text
候选码为 AC
```

BCNF 检查：

- `A -> B` 中 A 不是超码，违反 BCNF。
- `C -> D` 中 C 不是超码，违反 BCNF。
- `D -> E` 中 D 不是超码，违反 BCNF。
- `AC -> F` 中 AC 是候选码，满足 BCNF。

分解过程：

用 `A -> B` 分解：

```text
R1(A, B)
R2(A, C, D, E, F)
```

`R1` 中 A 是码，满足 BCNF。

在 `R2` 中，用 `C -> D` 分解：

```text
R3(C, D)
R4(A, C, E, F)
```

`R3` 中 C 是码，满足 BCNF。

在 `R4` 中，由于 `C -> D` 且 `D -> E`，可推出 `C -> E`，C 不是 `R4` 的超码，继续分解：

```text
R5(C, E)
R6(A, C, F)
```

`R5` 中 C 是码，满足 BCNF。`R6` 中 `AC -> F`，AC 是码，满足 BCNF。

最终 BCNF 分解：

```text
R1(A, B)
R3(C, D)
R5(C, E)
R6(A, C, F)
```

说明：也可以先用 `D -> E` 分解，得到的 BCNF 结果形式可能不同，只要过程正确、无损连接、各子关系满足 BCNF 即可。

## 五、并发控制与恢复题（10 分）

**题目：**

设调度 S：

```text
T1: r(X), w(X),          r(Y), w(Y), commit
T2:       r(Y),                  r(Z), commit
T3:             r(Z), w(Z), r(X), w(X), rollback
```

按时间顺序理解为：

```text
t1 T1 r(X)
t2 T2 r(Y)
t3 T3 r(Z)
t4 T1 w(X)
t5 T3 w(Z)
t6 T2 r(Z)
t7 T1 r(Y)
t8 T3 r(X)
t9 T1 w(Y)
t10 T3 w(X)
t11 T3 rollback
t12 T2 commit
t13 T1 commit
```

请判断是否冲突可串行化；说明是否可能死锁；指出 `T2` 在 t6 读取 `T3` 未提交写入的问题。

**答案：**

冲突边：

1. `T1` 写 `X` 在前，`T3` 读/写 `X` 在后：

```text
T1 -> T3
```

2. `T3` 写 `Z` 在前，`T2` 读 `Z` 在后：

```text
T3 -> T2
```

3. `T2` 读 `Y` 在前，`T1` 写 `Y` 在后：

```text
T2 -> T1
```

优先图中有环：

```text
T1 -> T3 -> T2 -> T1
```

所以该调度不是冲突可串行化调度。

若采用严格两段锁：

- `T1` 持有 `X` 的排他锁，后续需要 `Y` 的排他锁。
- `T2` 持有 `Y` 的共享锁，后续需要读 `Z`。
- `T3` 持有 `Z` 的排他锁，后续需要 `X` 的排他锁。

可能形成等待：

```text
T1 等 T2
T2 等 T3
T3 等 T1
```

因此可能发生死锁。

`T2` 在 t6 读取了 `T3` 已写但未提交的数据，而 `T3` 后来回滚。这属于脏读，也可能导致级联回滚。严格两段锁可以避免该问题，因为事务提交或回滚前不会释放排他锁。
