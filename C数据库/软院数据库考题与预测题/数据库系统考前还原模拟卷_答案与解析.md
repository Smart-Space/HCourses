### （二）简答题

视图是从一个或多个基本表或其他视图导出的虚表，其数据通常不独立完整存储，而由视图定义中的查询语句产生。

优点：可以简化复杂查询；可以屏蔽敏感字段，提高安全性；可以向不同用户提供不同的数据外模式，提高逻辑独立性。

局限：复杂视图可能带来查询性能开销；并非所有视图都可更新，尤其是包含聚合、分组、连接、去重等操作的视图。

```text
T_DB = π_sNo(σ_cName='数据库系统'(SC ⋈ Course))

π_sName(Student ⋈ (T_DB - T_OS))
```

解题要点：“选了 A 但没选 B”用集合差表达。

```text

π_sNo,sName(Student ⋈ T3)
```

解题要点：“选修了某集合中的全部课程”是除法运算的典型考法。

（9）

```sql
SELECT s.sNo, s.sName
FROM Student s
JOIN SC sc ON s.sNo = sc.sNo
JOIN Course c ON sc.cNo = c.cNo
GROUP BY s.sNo, s.sName
HAVING COUNT(*) = 1
   AND MAX(c.cName) = '算法';
```

解题要点：题目要求“只选修了算法一门课程”，可按学生分组，要求该学生选课记录总数为 1，且这一门课的课程名为“算法”。若数据库中可能存在重名课程，更严谨的写法应使用课程号判断：

```sql
SELECT s.sNo, s.sName
FROM Student s
JOIN SC sc ON s.sNo = sc.sNo
GROUP BY s.sNo, s.sName
HAVING COUNT(*) = 1
   AND MAX(sc.cNo) IN (
       SELECT cNo
       FROM Course
       WHERE cName = '算法'
   );
```

## 三、数据库设计题

### （1）ER 模型说明

实体及主要属性：

- 学院：学院编号、学院名称、办公地址、联系电话。
- 学生运动员：学号、姓名、性别、年龄。
- 比赛项目：项目编号、项目名称、项目类别、计分单位。
- 比赛：比赛编号、比赛时间、比赛阶段、天气。
- 场地：场地编号、场地名称、位置。
- 裁判：裁判编号、姓名、级别。

联系及基数：

- 学院与学生运动员：1:N。一个学院有多名学生运动员，每名学生运动员只属于一个学院。
- 比赛项目与比赛：1:N。一个项目可安排多场比赛，每场比赛属于一个项目。
- 场地与比赛：1:N。一个场地可承办多场比赛，每场比赛只在一个场地举行。
- 比赛与裁判：M:N。每场比赛有裁判参与，其中必须有一名主裁判；一名裁判可参与多场比赛。可在联系上设置角色属性。
- 学生运动员与比赛：M:N。学生报名参加比赛时，需要记录号码、道次、成绩、名次、积分，这些是参赛联系的属性。

### （2）关系模式转换

一种合理关系模式如下：

```text
Department(dNo, dName, addr, tel)
```

主码：`dNo`。

```text
Athlete(sNo, sName, sex, age, dNo)
```

主码：`sNo`。外码：`dNo` 参照 `Department(dNo)`。

```text
Event(eventNo, eventName, eventType, unit)
```

主码：`eventNo`。

```text
Venue(venueNo, venueName, location)
```

主码：`venueNo`。

```text
Judge(judgeNo, judgeName, judgeLevel)
```

主码：`judgeNo`。

```text
Competition(compNo, eventNo, venueNo, compTime, stage, weather)
```

主码：`compNo`。外码：`eventNo` 参照 `Event(eventNo)`，`venueNo` 参照 `Venue(venueNo)`。

```text
CompetitionJudge(compNo, judgeNo, role)
```

主码可取：`(compNo, judgeNo)`。外码：`compNo` 参照 `Competition(compNo)`，`judgeNo` 参照 `Judge(judgeNo)`。`role` 可取“主裁判”“普通裁判”。每场比赛恰有一名主裁判可通过唯一约束或业务规则保证。

```text
Participation(compNo, sNo, bibNo, laneNo, result, rankNo, points)
```

主码可取：`(compNo, sNo)`。外码：`compNo` 参照 `Competition(compNo)`，`sNo` 参照 `Athlete(sNo)`。

## 四、关系模式规范化题

已知：

```text
R(A, B, C, D, E, F, G, H)
F = { A -> BC, D -> EF, F -> G, AD -> H }
```

### （1）候选码

右部属性集合为：

```text
{B, C, E, F, G, H}
```

不出现在任何右部的属性为：

```text
{A, D}
```

因此任一候选码必须包含 `A` 和 `D`。

计算 `(AD)+`：

```text
AD
由 A -> BC 得 ABCD
由 D -> EF 得 ABCDEF
由 F -> G 得 ABCDEFG
由 AD -> H 得 ABCDEFGH
```

所以：

```text
候选码为 AD
```

### （2）BCNF 分解

BCNF 要求每个非平凡函数依赖 `X -> Y` 的左部 `X` 都是超码。

在原关系 R 中，候选码是 `AD`：

- `A -> BC` 中 A 不是超码，违反 BCNF。
- `D -> EF` 中 D 不是超码，违反 BCNF。
- `F -> G` 中 F 不是超码，违反 BCNF。
- `AD -> H` 中 AD 是候选码，满足 BCNF。

按 `A -> BC` 分解：

```text
R1(A, B, C)
R2(A, D, E, F, G, H)
```

`R1` 中 A 是码，满足 BCNF。

在 `R2` 中，`F -> G` 仍违反 BCNF，继续分解：

```text
R3(F, G)
R4(A, D, E, F, H)
```

`R3` 中 F 是码，满足 BCNF。

在 `R4` 中，`D -> EF` 仍违反 BCNF，继续分解：

```text
R5(D, E, F)
R6(A, D, H)
```

`R5` 中 D 是码，满足 BCNF。`R6` 中 AD 是码，满足 BCNF。

最终可得一组 BCNF 分解：

```text
R1(A, B, C)
R3(F, G)
R5(D, E, F)
R6(A, D, H)
```

## 五、可选考查题

### （1）优先图与冲突可串行化

分析冲突：

1. `T1` 在 `t4` 写 `X`，`T3` 在 `t8` 读 `X`、`t10` 写 `X`，得到边：

```text
T1 -> T3
```

2. `T3` 在 `t5` 写 `Z`，`T2` 在 `t6` 读 `Z`，得到边：

```text
T3 -> T2
```

3. `T2` 在 `t3` 读 `Y`，`T1` 在 `t9` 写 `Y`，得到边：

```text
T2 -> T1
```

优先图存在环：

```text
T1 -> T3 -> T2 -> T1
```

因此调度 S 不是冲突可串行化调度。

### （2）严格两段锁下的死锁分析

按访问顺序考虑锁：

1. `T1` 读写 `X`，需要持有 `X` 上的排他锁。
2. `T3` 读写 `Z`，需要持有 `Z` 上的排他锁。
3. `T2` 读 `Y`，需要持有 `Y` 上的共享锁。
4. `T2` 在 `t6` 想读 `Z`，会被 `T3` 持有的排他锁阻塞。
5. `T1` 在 `t7/t9` 想读写 `Y`，写 `Y` 时需要排他锁，会被 `T2` 的共享锁阻塞。
6. `T3` 在 `t8/t10` 想读写 `X`，会被 `T1` 的排他锁阻塞。

形成等待环：

```text
T1 等 T2，T2 等 T3，T3 等 T1
```

因此可能发生死锁。

### （3）读未提交数据的问题

`T2` 在 `t6` 读取了 `T3` 写入但尚未提交的数据，而 `T3` 在 `t11` 回滚。该问题属于脏读，也可能引发级联回滚。

可通过严格两段锁协议避免，因为严格两段锁要求事务提交或回滚前不释放排他锁，其他事务不能提前读到未提交写入。也可以通过恢复策略保证未提交更新不可被其他事务读取。
