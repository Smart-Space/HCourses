# 2015 年软微数据库系统考题解析

> 来源文件：`数据库15年软微.pdf`。该 PDF 为扫描件，主卷正文位于前 3 页，本解析按主卷题目整理。

## 一、简答题

### 1. 简述数据库、数据库管理系统和数据库系统的概念

**题目提取：** 简述数据库、数据库管理系统和数据库系统的概念。

**解题过程：**

1. 先区分数据对象、管理软件、完整运行环境三个层次。
2. 数据库是按一定数据模型组织、长期存储、可共享的数据集合。
3. DBMS 是位于用户与数据库之间的软件系统，负责定义、操纵、控制、维护数据库。
4. 数据库系统是由数据库、DBMS、数据库应用、数据库管理员、用户和相关软硬件环境组成的整体。

**参考答案：**

- 数据库（DB）：长期存储在计算机内、有组织、可共享的数据集合。
- 数据库管理系统（DBMS）：用于建立、使用和维护数据库的软件系统，提供数据定义、查询更新、事务管理、并发控制、安全控制和恢复等功能。
- 数据库系统（DBS）：以数据库为核心，由 DB、DBMS、应用程序、用户、DBA、计算机软硬件组成的系统。

### 2. 简述关系模型的特点

**题目提取：** 简述关系模型的特点。

**解题过程：**

1. 从数据结构、数据操作、完整性约束三个组成部分作答。
2. 说明关系模型用二维表表示数据，概念简单。
3. 强调集合操作、关系代数/关系演算基础，以及实体完整性、参照完整性、用户自定义完整性。

**参考答案：**

关系模型以关系即二维表作为基本数据结构，属性对应列，元组对应行；数据操作具有集合化、非过程化特点；支持实体完整性、参照完整性和用户自定义完整性；数据独立性较好，理论基础严密。

### 3. 简述事务的 ACID 特性

**题目提取：** 简述事务的 ACID 特性。

**解题过程：**

1. 将 ACID 展开为原子性、一致性、隔离性、持久性。
2. 每个性质说明其对事务正确性的作用。

**参考答案：**

- 原子性：事务中的操作要么全部完成，要么全部不做。
- 一致性：事务执行前后数据库都应满足完整性约束。
- 隔离性：并发事务之间互不干扰，效果等价于某种串行执行。
- 持久性：事务提交后，其结果即使系统故障也不会丢失。

### 4. 简述视图概念以及通过视图访问数据的优缺点

**题目提取：** 简述视图概念以及通过视图访问数据库中数据的优缺点。

**解题过程：**

1. 先定义视图是从基本表或其他视图导出的虚表。
2. 优点从简化查询、安全保护、逻辑独立性角度说明。
3. 缺点从更新限制、性能开销、依赖基本表角度说明。

**参考答案：**

视图是一个虚表，其内容由查询定义，不独立存储完整数据。优点是可以简化复杂查询、屏蔽敏感字段、提高数据逻辑独立性、从不同角度组织数据。缺点是有些视图不可更新，复杂视图查询可能带来性能开销，且视图依赖底层基本表。

### 5. 简述应用程序通过 JDBC 方式连接数据库的过程

**题目提取：** 简述应用程序通过 JDBC 方式连接数据库的过程。

**解题过程：**

1. 准备 JDBC 驱动。
2. 加载或注册驱动。
3. 通过 URL、用户名、密码建立连接。
4. 创建 Statement/PreparedStatement。
5. 执行 SQL、处理 ResultSet。
6. 关闭结果集、语句和连接。

**参考答案：**

加载 JDBC 驱动，调用 `DriverManager.getConnection(url, user, password)` 获取连接，创建 SQL 执行对象，执行查询或更新，处理返回结果，最后释放 `ResultSet`、`Statement`、`Connection` 等资源。

## 二、数据查询题

### 已知关系模式

```text
Department(dNo, dName, addr, tel)
Student(sNo, sName, sex, age, dNo)
Course(cNo, cName, type, credit, dNo)
CourseRel(cNo, cPNo)
SC(sNo, cNo, score)
```

其中：

- `Department.dNo` 为主码。
- `Student.sNo` 为主码，`Student.dNo` 参照 `Department.dNo`。
- `Course.cNo` 为主码，`Course.dNo` 参照 `Department.dNo`。
- `CourseRel(cNo, cPNo)` 表示课程与先修课程，二者均参照 `Course.cNo`。
- `SC(sNo, cNo)` 为主码，分别参照 `Student.sNo` 和 `Course.cNo`。

### 1. 用关系代数表达式完成查询

#### （1）查询“数据库系统”课程成绩小于 60 分的“软件学院”所有学生姓名

**解题过程：**

1. 学生姓名在 `Student`。
2. 学院名称在 `Department`。
3. 课程名在 `Course`。
4. 成绩在 `SC`。
5. 四表连接后筛选学院、课程、成绩，再投影姓名。

**答案：**

```text
π_sName(
  σ_dName='软件学院' ∧ cName='数据库系统' ∧ score<60
  (Department ⋈ Student ⋈ SC ⋈ Course)
)
```

如果原题学院名为“软件与微电子学院”，将筛选条件中的学院名替换为该名称即可。

#### （2）查询选修了“软件学院”所开设全部课程的学生姓名

**解题过程：**

1. 先求软件学院开设的全部课程号集合。
2. 再求学生选课二元组 `(sNo, cNo)`。
3. 使用除法查询“对课程集合中每一门课都选了”的学生。
4. 与 `Student` 连接得到姓名。

**答案：**

```text
T1 = π_cNo(σ_dName='软件学院'(Department ⋈ Course))
T2 = π_sNo,cNo(SC)
T3 = T2 ÷ T1
π_sName(Student ⋈ T3)
```

#### （3）查询没有选修课程名为“数据库系统”的所有学生姓名

**解题过程：**

1. 先求选修了“数据库系统”的学生号。
2. 用全体学生号减去这些学生号。
3. 与 `Student` 连接并投影姓名。

**答案：**

```text
T1 = π_sNo(σ_cName='数据库系统'(SC ⋈ Course))
π_sName(Student ⋈ (π_sNo(Student) - T1))
```

### 2. 用 SQL 语句完成查询和更新

#### （1）CourseRel 创建时未添加主码和外键，创建后补充约束

**解题过程：**

1. `CourseRel` 表示课程与先修课程。
2. 一门课程可有多个先修课程，因此联合主码为 `(cNo, cPNo)`。
3. `cNo` 和 `cPNo` 都应引用 `Course(cNo)`。

**答案：**

```sql
ALTER TABLE CourseRel
ADD CONSTRAINT pk_CourseRel PRIMARY KEY (cNo, cPNo);

ALTER TABLE CourseRel
ADD CONSTRAINT fk_CourseRel_course
FOREIGN KEY (cNo) REFERENCES Course(cNo);

ALTER TABLE CourseRel
ADD CONSTRAINT fk_CourseRel_prereq
FOREIGN KEY (cPNo) REFERENCES Course(cNo);
```

#### （2）将“软件学院”所有学生“数据库系统”课程成绩增加 10 分

**解题过程：**

1. 被更新表是 `SC`。
2. 限定条件需要关联学生所在学院和课程名称。
3. 用子查询筛选满足条件的选课记录。

**答案：**

```sql
UPDATE SC
SET score = score + 10
WHERE cNo IN (
    SELECT c.cNo
    FROM Course c
    WHERE c.cName = '数据库系统'
)
AND sNo IN (
    SELECT s.sNo
    FROM Student s
    JOIN Department d ON s.dNo = d.dNo
    WHERE d.dName = '软件学院'
);
```

#### （3）查询选修人数大于 100 人的课程名称及开课学院名称

**解题过程：**

1. 按课程分组统计选课人数。
2. `HAVING COUNT(*) > 100` 保留人数超过 100 的课程。
3. 连接 `Course` 与 `Department` 输出课程名、学院名。

**答案：**

```sql
SELECT c.cName, d.dName
FROM Course c
JOIN Department d ON c.dNo = d.dNo
JOIN SC sc ON c.cNo = sc.cNo
GROUP BY c.cNo, c.cName, d.dName
HAVING COUNT(*) > 100;
```

#### （4）查询选修了课程名为“A”但没选修课程名为“B”的学生人数

**解题过程：**

1. 先找选修 A 的学生。
2. 排除选修 B 的学生。
3. 对剩余学生去重计数。

**答案：**

```sql
SELECT COUNT(DISTINCT sca.sNo) AS student_count
FROM SC sca
JOIN Course ca ON sca.cNo = ca.cNo
WHERE ca.cName = 'A'
AND NOT EXISTS (
    SELECT 1
    FROM SC scb
    JOIN Course cb ON scb.cNo = cb.cNo
    WHERE scb.sNo = sca.sNo
      AND cb.cName = 'B'
);
```

#### （5）查询选修了课程名为“DS”但成绩小于 60 或成绩为空的学生姓名

**解题过程：**

1. 连接学生、选课、课程。
2. 限定课程名为 DS。
3. 成绩条件为 `<60 OR IS NULL`。

**答案：**

```sql
SELECT s.sName
FROM Student s
JOIN SC sc ON s.sNo = sc.sNo
JOIN Course c ON sc.cNo = c.cNo
WHERE c.cName = 'DS'
  AND (sc.score < 60 OR sc.score IS NULL);
```

#### （6）查询课程名为“DB”的先修课程名称

**解题过程：**

1. 先由课程名 DB 找到课程号。
2. 在 `CourseRel` 中找到其先修课程号 `cPNo`。
3. 再连接 `Course` 得到先修课程名称。

**答案：**

```sql
SELECT p.cName
FROM Course db
JOIN CourseRel r ON db.cNo = r.cNo
JOIN Course p ON r.cPNo = p.cNo
WHERE db.cName = 'DB';
```

#### （7）查询“胡图图”同学选修课程的总学分

**解题过程：**

1. 由学生姓名找到学生。
2. 连接选课表和课程表。
3. 对所选课程学分求和。

**答案：**

```sql
SELECT SUM(c.credit) AS total_credit
FROM Student s
JOIN SC sc ON s.sNo = sc.sNo
JOIN Course c ON sc.cNo = c.cNo
WHERE s.sName = '胡图图';
```

## 三、分析题

### 1. 并发调度分析

**题目提取：** 给出事务 `Ta`、`Tb`、`Tc` 的交叉执行序列，要求判断调度是否冲突可串行化；若采用两段锁协议，给出相应加锁过程，并判断能否顺利执行。

**关键调度：**

```text
Ta: read(x), x=x-10, write(x), read(y), y=y+10, write(y), commit
Tb: read(y), s=s+y, read(z), s=s+z, commit
Tc: read(z), z=z+100, write(z), read(x), x=x+100, write(x), rollback
```

**（1）冲突可串行化判断过程：**

1. `Ta` 先写 `x`，`Tc` 后读/写 `x`，得到边 `Ta -> Tc`。
2. `Tc` 先写 `z`，`Tb` 后读 `z`，得到边 `Tc -> Tb`。
3. `Tb` 先读 `y`，`Ta` 后写 `y`，得到边 `Tb -> Ta`。
4. 优先图中存在环：`Ta -> Tc -> Tb -> Ta`。

**答案：** 该调度不是冲突可串行化调度。

**（2）采用两段锁协议的加锁过程：**

一种可能的专用锁加锁过程如下：

```text
Ta: XLOCK(x) -> read/write x -> XLOCK(y) -> read/write y -> UNLOCK(x,y)
Tb: SLOCK(y) -> read y -> SLOCK(z) -> read z -> UNLOCK(y,z)
Tc: XLOCK(z) -> read/write z -> XLOCK(x) -> read/write x -> rollback -> UNLOCK(z,x)
```

若严格按原调度时间推进：

1. `Ta` 持有 `X(x)`。
2. `Tc` 持有 `X(z)`。
3. `Tb` 持有 `S(y)`。
4. `Ta` 申请 `X(y)`，被 `Tb` 的 `S(y)` 阻塞。
5. `Tb` 申请 `S(z)`，被 `Tc` 的 `X(z)` 阻塞。
6. `Tc` 申请 `X(x)`，被 `Ta` 的 `X(x)` 阻塞。

**答案：** 会形成死锁，不能顺利执行，需要死锁检测、超时回滚或改变调度顺序。

### 2. 局部 ER 模型问题分析

**题目提取：** 给出 `A - B - C` 的局部 ER 模型，要求分析可能存在的问题并给出修正方法。

**解题过程：**

1. 检查每个实体参与联系的最小基数和最大基数是否符合业务含义。
2. 检查是否存在隐含的传递依赖或应合并为三元联系的情况。
3. 检查两个二元联系是否会丢失 `A`、`B`、`C` 同时参与同一业务事实的信息。

**参考答案：**

若业务事实本身是“某个 A 通过某个 B 对某些 C 发生联系”，则拆成 `A-B` 和 `B-C` 两个二元联系会丢失 A 与 C 的配对语义，可能产生错误组合。修正方法是改为三元联系 `R(A,B,C)`，并在三元联系上标注正确的参与约束；若 `B` 只是连接实体，则也可将 `B` 设计为联系实体，并保留能唯一标识业务事实的主码。

## 四、设计题

### 球赛信息管理系统数据库设计

**题目提取：**

已知实体及属性：

- 球队：球队编号、名称、荣誉、积分。
- 球员：球员编号、姓名、年龄、国籍。
- 教练：教练编号、姓名、年龄、国籍。
- 赛场：场地编号、名称、地址。

约束：

- 每支球队必须有一名主教练，一名教练只能担任一支球队的主教练。
- 每支球队有多名球员，一名球员只能属于一支球队。
- 球队之间按照赛程在既定时间和赛场举行比赛，每次比赛产生一个比分。

要求构造 ER 模型，并转换为关系模式。

**解题过程：**

1. 识别强实体：球队、球员、教练、赛场、比赛。
2. 球队与教练是 1:1 联系，可把主教练编号作为球队外键。
3. 球队与球员是 1:N 联系，可把球队编号作为球员外键。
4. 比赛涉及两支球队、一个赛场、比赛时间和比分，应单独建比赛表。
5. 为避免主队和客队混淆，比赛表中设置 `team1No`、`team2No` 或 `homeTeamNo`、`awayTeamNo`。

**关系模式：**

```text
Coach(coachNo, coachName, age, nationality)
Team(teamNo, teamName, honor, points, coachNo)
Player(playerNo, playerName, age, nationality, teamNo)
Venue(venueNo, venueName, address)
Match(matchNo, matchTime, venueNo, homeTeamNo, awayTeamNo, homeScore, awayScore)
```

**主外键说明：**

- `Coach.coachNo` 为主码。
- `Team.teamNo` 为主码，`Team.coachNo` 为外键并加唯一约束，参照 `Coach.coachNo`。
- `Player.playerNo` 为主码，`Player.teamNo` 参照 `Team.teamNo`。
- `Venue.venueNo` 为主码。
- `Match.matchNo` 为主码，`venueNo` 参照 `Venue`，`homeTeamNo` 和 `awayTeamNo` 参照 `Team`。

## 五、关系模式规范化

### 题目提取

设关系模式：

```text
R(A, B, C, D, E, F, G, H)
```

函数依赖集扫描件可辨识为：

```text
F = { A -> BH, C -> DF, E -> F, AC -> G }
```

> 注：扫描件该行较模糊，若教师原卷中依赖与此不同，应按原依赖重新计算闭包、候选码和 BCNF 分解。

要求：

1. 求最小函数依赖集。
2. 求 R 的候选码。
3. 将 R 逐步分解为满足 BCNF 的关系模式，并写出规范化过程。

### （1）求最小函数依赖集

**解题过程：**

1. 右部拆成单属性：
   `A->B`、`A->H`、`C->D`、`C->F`、`E->F`、`AC->G`。
2. 检查左部是否有冗余属性。只有 `AC->G` 左部含两个属性。
3. 计算 `A+` 不能推出 `G`，`C+` 也不能推出 `G`，因此 `AC->G` 左部无冗余属性。
4. 检查依赖是否冗余，逐条删除后均不能由其余依赖推出。

**答案：**

```text
Fm = { A->B, A->H, C->D, C->F, E->F, AC->G }
```

### （2）求候选码

**解题过程：**

1. 不出现在任何依赖右部的属性必须包含在候选码中。
2. 右部属性有 `B,H,D,F,G`，所以 `A,C,E` 必须出现在任一候选码中。
3. 计算 `(ACE)+`：
   - 初始 `{A,C,E}`。
   - 由 `A->BH` 得 `{A,B,C,E,H}`。
   - 由 `C->DF` 得 `{A,B,C,D,E,F,H}`。
   - 由 `AC->G` 得 `{A,B,C,D,E,F,G,H}`。
4. `ACE` 能推出全部属性，且 A、C、E 都不能删除。

**答案：**

```text
候选码：{A, C, E}
```

### （3）BCNF 分解

**解题过程：**

BCNF 要求每个非平凡函数依赖 `X->Y` 中，`X` 必须是超码。原关系中 `A`、`C`、`E`、`AC` 均不是 R 的超码，因此需要分解。

一种可行分解过程：

1. 用 `A->BH` 分解：

```text
R1(A, B, H)
R2(A, C, D, E, F, G)
```

2. 在 `R2` 中用 `C->DF` 分解：

```text
R21(C, D, F)
R22(A, C, E, G)
```

3. 在 `R22` 中用 `AC->G` 分解：

```text
R221(A, C, G)
R222(A, C, E)
```

4. 依赖 `E->F` 未被保留在上述分解中。若要求尽量保留依赖，可补充分解关系：

```text
R3(E, F)
```

**BCNF 关系组：**

```text
R1(A, B, H)
R21(C, D, F)
R221(A, C, G)
R222(A, C, E)
R3(E, F)
```

**说明：**

- `R1` 中 A 是码，满足 BCNF。
- `R21` 中 C 是码，满足 BCNF。
- `R221` 中 AC 是码，满足 BCNF。
- `R222` 无非平凡依赖，满足 BCNF。
- `R3` 中 E 是码，满足 BCNF。
