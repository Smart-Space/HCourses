# 2024 数据库系统考题解析

> 来源文件：`2024数据库系统考题.docx`。原文件是考题回忆提纲，部分题目写有“同 15 年”“查询内容略有变化”，因此本解析对完整题干缺失处按可判定信息和 2015 年同题结构补全，并对无法唯一确定的题目明确标注。

## 一、填空题（30 分）

### 1. 关系模型的三个完整性约束

**题目提取：** 关系模型的三个完整性约束：1__ 2__ 3__。

**答案：**

```text
实体完整性、参照完整性、用户自定义完整性
```

**解题过程：**

关系模型由关系数据结构、关系操作集合、关系完整性约束三部分组成。完整性约束通常分为实体完整性、参照完整性和用户自定义完整性。

### 2. 关系模型操作后得到的数据结构

**题目提取：** 关系模型操作后得到的数据结构是__。

**答案：**

```text
关系
```

**解题过程：**

关系模型具有闭包性，关系代数运算的输入是关系，输出仍然是关系。

### 3. 数据库设计三个阶段的核心阶段

**题目提取：** 数据库设计的三个阶段的核心阶段是__。

**答案：**

```text
概念结构设计
```

**解题过程：**

若按概念设计、逻辑设计、物理设计三阶段划分，概念结构设计负责抽象现实世界数据需求并形成 ER 模型，是数据库设计的关键和核心。

### 4. JDBC 连接数据库时需要的参数

**题目提取：** JDBC 连接数据库时需要的参数有 1__ 2__ 3__。

**答案：**

```text
数据库 URL、用户名、密码
```

**解题过程：**

典型调用是 `DriverManager.getConnection(url, user, password)`。若题目把驱动也计入连接准备过程，则还应写 JDBC 驱动类名。

### 5. 分解到第几范式可消除大部分异常

**题目提取：** 分解到第__范式消除大部分异常。

**答案：**

```text
第三范式（3NF）
```

**解题过程：**

2NF 消除非主属性对候选码的部分函数依赖，3NF 进一步消除非主属性对候选码的传递依赖，通常认为可消除大部分插入、删除和更新异常。

### 6. 严格二段锁是冲突可串行化的什么条件

**题目提取：** 严格二段锁是冲突什么什么并行的__条件。

**答案：**

```text
充分条件
```

**解题过程：**

遵守二段锁协议的调度一定是冲突可串行化的，因此二段锁是冲突可串行化的充分条件，不是必要条件。严格二段锁还保证可恢复性并避免级联回滚。

### 7. 系统故障时 DBMS 执行的操作

**题目提取：** DBMS 是非强制返回，所以当系统出现故障时 DBMS 执行__操作。

**答案：**

```text
恢复操作
```

**解题过程：**

系统故障后，DBMS 根据日志执行恢复：对已提交事务重做，对未提交事务撤销。

### 8. 对同一关系模式可用什么索引

**题目提取：** 对同一关系模式可用__索引（原文带问号，题干不完整）。

**参考答案：**

```text
多个辅助索引
```

**解题过程：**

同一关系只能按一种物理顺序组织，因此通常只有一个聚簇索引；但可建立多个非聚簇/辅助索引以支持不同查询条件。原题缺少完整语境，若课程使用的表述为“可有多个索引”，也可答“多个”。

### 9. 操作顺序的优化

**题目提取：** 操作顺序的优化。

**参考答案：**

```text
选择、投影尽早做；笛卡尔积与选择合并为连接；优先执行能显著减少中间结果的连接。
```

**解题过程：**

关系查询优化的启发式规则是尽早缩小元组数和属性数，减少中间结果规模。

### 10. 图数据库使用的语言

**题目提取：** 图数据库使用的语言是__。

**答案：**

```text
Cypher
```

**解题过程：**

常见图数据库 Neo4j 使用 Cypher 作为图查询语言。

## 二、SQL 题（40 分）

原文件说明：

```text
同15年
查询内容略有变化
sql（1）（2）完全一致
增加了视图（view）的考察
```

因此这里保留 2015 年同结构关系模式，并补充视图题。

### 基本关系模式

```text
Department(dNo, dName, addr, tel)
Student(sNo, sName, sex, age, dNo)
Course(cNo, cName, type, credit, dNo)
CourseRel(cNo, cPNo)
SC(sNo, cNo, score)
```

### 1. 创建后补充 CourseRel 的主键和外键

**答案：**

```sql
ALTER TABLE CourseRel
ADD CONSTRAINT pk_CourseRel PRIMARY KEY (cNo, cPNo);

ALTER TABLE CourseRel
ADD CONSTRAINT fk_CourseRel_course
FOREIGN KEY (cNo) REFERENCES Course(cNo);

ALTER TABLE CourseRel
ADD CONSTRAINT fk_CourseRel_prereq
FOREIGN KEY (cPNo) REFERENCES Course(cNo);
```

**解题过程：**

`CourseRel` 是课程和先修课程的联系表，联合主码应为 `(cNo, cPNo)`，两个字段均引用课程表主码。

### 2. 更新“软件学院”学生“数据库系统”课程成绩

**答案：**

```sql
UPDATE SC
SET score = score + 10
WHERE cNo IN (
    SELECT c.cNo
    FROM Course c
    WHERE c.cName = '数据库系统'
)
AND sNo IN (
    SELECT s.sNo
    FROM Student s
    JOIN Department d ON s.dNo = d.dNo
    WHERE d.dName = '软件学院'
);
```

**解题过程：**

更新目标是 `SC.score`，但筛选条件来自课程名和学院名，因此用子查询或连接限定对应选课记录。

### 3. 视图考察：创建学生成绩视图

**可能题目提取：** 增加视图（view）的考察。可按常见题型理解为创建一个包含学生、课程、成绩的视图，并基于视图查询。

**答案示例：**

```sql
CREATE VIEW v_student_course_score AS
SELECT
    s.sNo,
    s.sName,
    d.dName,
    c.cNo,
    c.cName,
    sc.score
FROM Student s
JOIN Department d ON s.dNo = d.dNo
JOIN SC sc ON s.sNo = sc.sNo
JOIN Course c ON sc.cNo = c.cNo;
```

基于视图查询不及格学生：

```sql
SELECT sName, cName, score
FROM v_student_course_score
WHERE score < 60;
```

**解题过程：**

视图题通常考 `CREATE VIEW 视图名 AS SELECT ...`。先写出视图需要暴露的列，再完成多表连接，最后可像普通表一样查询视图。

## 三、ER 图设计题（20 分）

### 题目提取

餐厅点菜系统：

- 一个顾客分配一个餐桌号。
- 一个餐桌由一个服务员负责。
- 顾客用菜单进行点餐。
- 点餐后每个菜品由一个厨师制作。
- 系统需要保存餐桌、服务员、服务、菜品、点餐、厨师的数据。

### 解题过程

1. 识别实体：顾客、餐桌、服务员、菜单、菜品、点餐、厨师。
2. 顾客与餐桌：一个顾客分配一个餐桌；一张餐桌在同一时段可对应一个或多个顾客，视业务可建分配关系。
3. 餐桌与服务员：一个餐桌由一个服务员负责，一个服务员可负责多张餐桌。
4. 顾客与菜单/点餐：顾客通过菜单点餐，点餐单应作为独立实体。
5. 点餐与菜品：一张点餐单包含多个菜品，一个菜品可出现在多张点餐单中，因此是 M:N，需建明细表。
6. 菜品与厨师：题目说“点餐后每个菜品由一个厨师制作”，更准确的是点餐明细由厨师制作，因为同一菜品不同订单可能由不同厨师完成。

### 关系模式答案

```text
Customer(customerNo, customerName, phone)
DiningTable(tableNo, capacity, status)
Waiter(waiterNo, waiterName, phone)
Menu(menuNo, menuName)
Dish(dishNo, dishName, price, menuNo)
Chef(chefNo, chefName, phone)
TableService(tableNo, waiterNo, serviceDate)
DiningOrder(orderNo, customerNo, tableNo, orderTime)
OrderItem(orderNo, dishNo, quantity, chefNo, itemStatus)
```

### 主外键说明

- `Customer.customerNo`、`DiningTable.tableNo`、`Waiter.waiterNo`、`Menu.menuNo`、`Dish.dishNo`、`Chef.chefNo`、`DiningOrder.orderNo` 为主码。
- `Dish.menuNo` 参照 `Menu.menuNo`。
- `TableService(tableNo, waiterNo, serviceDate)` 可用 `(tableNo, serviceDate)` 作主码，分别参照餐桌和服务员。
- `DiningOrder.customerNo` 参照顾客，`DiningOrder.tableNo` 参照餐桌。
- `OrderItem(orderNo, dishNo)` 为主码，`chefNo` 参照厨师。

## 四、范式题

### 题目提取

原文件只写明：

```text
（1）求候选码
（2）分解为BCNF范式
```

但没有给出关系模式和函数依赖集，因此不能得到唯一数值答案。以下给出标准解题过程。

### 1. 求候选码的过程

1. 列出关系模式 `R(U)` 的全部属性集合 `U`。
2. 列出函数依赖集 `F`。
3. 找出不出现在任何依赖右部的属性，这些属性必须包含在任一候选码中。
4. 对必要属性集求闭包。
5. 若闭包不能覆盖 `U`，逐个加入可能属性继续求闭包。
6. 能推出全部属性且没有冗余属性的最小属性集就是候选码。

**书写模板：**

```text
设必要属性集为 X。
X+ = {...}
若 X+ = U，则 X 为候选码。
若 X+ != U，则尝试 XA、XB ...
最后删除冗余属性，得到全部候选码。
```

### 2. BCNF 分解过程

1. 判断每个非平凡函数依赖 `X -> Y`。
2. 若 `X` 不是当前关系的超码，则违反 BCNF。
3. 按该依赖分解：

```text
R 分解为 R1 = X ∪ Y，R2 = R - (Y - X)
```

4. 对 `R1` 和 `R2` 分别投影函数依赖并继续判断。
5. 直到每个子关系中所有非平凡依赖的左部都是超码。

**答案模板：**

```text
因为 X -> Y 且 X 不是 R 的超码，所以 R 不满足 BCNF。
分解：
R1(X, Y)
R2(R - Y + X)
继续检查 R1、R2。
最终得到：R1(...), R2(...), ...
```
