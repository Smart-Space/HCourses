## 题目 2：每个范式的代价和作用

**题目提取：** 每个范式的代价和作用。

### 1NF

**作用：** 保证属性值不可再分，消除重复组，使关系成为规范二维表。

**代价：** 需要拆分复合属性或多值属性，表数量和连接需求可能增加。

### 2NF

**作用：** 在 1NF 基础上，消除非主属性对候选码的部分函数依赖，主要针对组合码关系。

**代价：** 需要拆分表，查询完整信息时可能需要连接。

### 3NF

**作用：** 在 2NF 基础上，消除非主属性对候选码的传递函数依赖，减少数据冗余和更新异常。

**代价：** 表进一步拆分，查询路径变长，连接开销增加。

### BCNF

**作用：** 比 3NF 更严格，要求每个非平凡函数依赖的决定因素都是超码，进一步消除由非码决定因素引起的异常。

**代价：** BCNF 分解可能不保持函数依赖，需要通过额外约束或应用逻辑保证某些依赖。

### 4NF

**作用：** 消除非平凡多值依赖导致的冗余。

**代价：** 关系继续拆分，查询时需要更多连接。

### 解题总结

规范化程度越高，冗余和更新异常越少，但表数量和连接成本通常增加。实际设计需要在数据一致性和查询效率之间权衡。

## 题目 3：逻辑模型设计的意义

**题目提取：** 逻辑模型设计的意义。

**解题过程：**

1. 概念模型通常是 ER 模型，不能直接被关系数据库实现。
2. 逻辑模型设计把概念结构转换为特定 DBMS 支持的数据模型。
3. 在关系数据库中，逻辑设计结果通常是关系模式、主码、外码和完整性约束。

**参考答案：**

逻辑模型设计的意义在于把独立于具体 DBMS 的概念模型转换为某一数据模型下的逻辑结构，使数据库能够被具体 DBMS 实现；同时通过主码、外码、约束和规范化设计降低冗余、减少异常、保证数据一致性。

## 题目 4：两段锁协议的意义

**题目提取：** 两段锁协议的意义。

**解题过程：**

1. 两段锁协议把事务分为加锁阶段和解锁阶段。
2. 加锁阶段只能申请锁，不能释放锁。
3. 一旦释放锁，就进入解锁阶段，不能再申请新锁。
4. 该协议用于保证并发调度的冲突可串行化。

**参考答案：**

两段锁协议是并发控制协议，能够保证遵守协议的调度是冲突可串行化调度，从而保持并发事务执行结果的正确性。严格两段锁还要求事务提交或回滚前不释放排他锁，可避免级联回滚并保证可恢复性。

## 题目 5：各个属性属于什么码

**题目提取：** 各个属性属于什么码。

原文件没有给出具体关系模式和函数依赖，因此不能计算具体属性归类。以下是解题模板。

### 解题过程

1. 根据函数依赖求候选码。
2. 出现在任一候选码中的属性称为主属性。
3. 不出现在任何候选码中的属性称为非主属性。
4. 从候选码中选定一个作为主码。
5. 若某属性或属性组引用其他关系的主码，则为外码。
6. 能唯一标识元组但未被选为主码的候选码，可称为候选码或备用码。

### 书写模板

```text
候选码：K1, K2
主码：K1
主属性：出现在 K1 或 K2 中的属性
非主属性：其余属性
外码：参照其他关系主码的属性
```

## 题目 6：SQL 题

**题目提取：** SQL 考的是增加视图、更改数据、在创建表后补充外键和主键。

由于 2021 回忆卷没有给出表结构，以下按 2015/2024 同类数据库课程常见表结构给出写法。

### 1. 增加视图

**通用答案：**

```sql
CREATE VIEW v_student_score AS
SELECT
    s.sNo,
    s.sName,
    c.cNo,
    c.cName,
    sc.score
FROM Student s
JOIN SC sc ON s.sNo = sc.sNo
JOIN Course c ON sc.cNo = c.cNo;
```

**解题过程：**

先明确视图要暴露的字段，再把这些字段所在的基本表连接起来。视图语法是 `CREATE VIEW 视图名 AS SELECT ...`。

### 2. 更改数据

**通用答案示例：**

```sql
UPDATE SC
SET score = score + 10
WHERE cNo = (
    SELECT cNo
    FROM Course
    WHERE cName = '数据库系统'
);
```

**解题过程：**

先找被修改字段所在表，再把题目中的业务条件转换为 `WHERE` 子句。涉及其他表的条件用子查询或连接实现。

### 3. 创建表后补充主键和外键

**通用答案示例：**

```sql
ALTER TABLE CourseRel
ADD CONSTRAINT pk_CourseRel PRIMARY KEY (cNo, cPNo);

ALTER TABLE CourseRel
ADD CONSTRAINT fk_CourseRel_course
FOREIGN KEY (cNo) REFERENCES Course(cNo);

ALTER TABLE CourseRel
ADD CONSTRAINT fk_CourseRel_prereq
FOREIGN KEY (cPNo) REFERENCES Course(cNo);
```

**解题过程：**

主键约束用 `PRIMARY KEY`，外键约束用 `FOREIGN KEY ... REFERENCES ...`。联系表通常使用参与联系的两个外键组成联合主键。

## 题目 7：足球比赛 ER 设计题

**题目提取：** 大题和老卷子一样考的图一样，增加了关系：球队可以派遣球员上场，但需要记录比赛数据。

### 解题过程

1. 识别实体：球队、球员、教练、赛场、比赛。
2. 老卷约束通常包括：球队有主教练，球队有球员，球队之间在赛场比赛并产生比分。
3. 新增关系是球队派遣球员上场并记录比赛数据，因此需要为“球员参加比赛”单独建联系表。
4. 比赛数据不能只放在球员表或球队表，因为同一球员可参加多场比赛，同一比赛也有多名球员上场。

### 关系模式答案

```text
Coach(coachNo, coachName, age, nationality)
Team(teamNo, teamName, honor, points, coachNo)
Player(playerNo, playerName, age, nationality, teamNo)
Venue(venueNo, venueName, address)
Match(matchNo, matchTime, venueNo, homeTeamNo, awayTeamNo, homeScore, awayScore)
Appearance(matchNo, teamNo, playerNo, isStarter, playMinutes, goals, assists, yellowCards, redCards)
```

### 主外键说明

- `Team.coachNo` 参照 `Coach.coachNo`，并应加唯一约束以保证一名教练只担任一支球队主教练。
- `Player.teamNo` 参照 `Team.teamNo`。
- `Match.venueNo` 参照 `Venue.venueNo`。
- `Match.homeTeamNo` 和 `Match.awayTeamNo` 参照 `Team.teamNo`。
- `Appearance(matchNo, playerNo)` 可作主码。
- `Appearance.matchNo` 参照 `Match`，`Appearance.teamNo` 参照 `Team`，`Appearance.playerNo` 参照 `Player`。

### 关键约束

```text
homeTeamNo <> awayTeamNo
Appearance.teamNo 必须等于该场比赛的 homeTeamNo 或 awayTeamNo
Appearance.playerNo 所属球队应与 Appearance.teamNo 一致
```

这些复杂约束通常不能只靠普通外键完全表达，可通过触发器、断言或应用层校验实现。

## 题目 8：BCNF 范式题

**题目提取：** 最后一题 BCNF 范式，要求给出过程。

原文件没有给出具体关系模式和函数依赖，因此不能给出唯一分解结果。以下是标准过程。

### 解题过程

1. 求候选码：

```text
对可能的属性集 X 求闭包 X+。
若 X+ 包含关系的全部属性，则 X 是超码。
若 X 是超码且任一真子集都不是超码，则 X 是候选码。
```

2. 检查 BCNF：

```text
对每个非平凡函数依赖 X -> Y：
若 X 不是超码，则违反 BCNF。
```

3. 分解违反 BCNF 的关系：

```text
R1 = X ∪ Y
R2 = R - (Y - X)
```

4. 递归检查 `R1` 和 `R2`，直到所有子关系满足 BCNF。

### 答题模板

```text
已知 R(...)，F={...}

1. 求闭包：
X+ = ...
所以候选码为 ...

2. 检查依赖：
因为 X -> Y 中 X 不是超码，所以 R 不满足 BCNF。

3. 分解：
R1(X, Y)
R2(...)

4. 继续检查：
R1 满足 BCNF；
R2 若仍有违反项，继续分解。

最终结果：
R1(...), R2(...), ...
```
