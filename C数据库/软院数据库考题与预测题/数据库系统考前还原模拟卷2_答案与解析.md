### （二）简答题

事务并发执行可能带来三类典型问题：

- 丢失修改：两个事务同时修改同一数据，一个事务的修改覆盖另一个事务的修改。
- 不可重复读：同一事务两次读取同一数据，中间被其他事务修改并提交，导致两次读取结果不同。
- 读脏数据：一个事务读取了另一个尚未提交事务写入的数据，而后者后来回滚。

并发控制的目标是保证多个事务并发执行时，执行结果等价于某种串行执行，从而保持数据库一致性，通常用可串行化作为正确性准则。

CREATE VIEW v_course_stat AS
SELECT
    c.cNo,
    c.cName,
    COUNT(*) AS studentCount,
    AVG(sc.score) AS avgScore
FROM Course c
JOIN SC sc ON c.cNo = sc.cNo
GROUP BY c.cNo, c.cName;

说明：`AVG` 会自动忽略空值；若课程有选课记录但所有成绩为空，则平均成绩为空。

（9）

```sql
SELECT
    s.sNo,
    s.sName,
    AVG(sc.score) AS avgScore
FROM Student s
JOIN SC sc ON s.sNo = sc.sNo
WHERE sc.score IS NOT NULL
GROUP BY s.sNo, s.sName
HAVING COUNT(*) >= 3
   AND AVG(sc.score) >= 85;
```

说明：如果题目要求“至少选修 3 门课程”包含成绩为空的课程，则应把 `COUNT(*)` 放在不过滤空成绩的选课记录上；本题默认平均成绩只统计已录入成绩。

## 三、数据库设计题

### （1）ER 模型说明

实体及主要属性：

- 读者：读者编号、姓名、类型、联系电话。
- 图书：图书编号、书名、作者、出版社、出版年份、分类号。
- 馆藏副本：副本编号、入馆日期、存放位置、状态。
- 馆员：馆员编号、姓名、岗位。

联系及基数：

- 图书与馆藏副本：1:N。一本图书可以有多册馆藏副本，每个副本只属于一本图书。
- 读者与馆藏副本之间通过“借阅”联系形成 M:N。联系属性包括借阅时间、应还时间、实际归还时间、续借次数。
- 馆员与借阅业务存在办理关系。一次借阅需要记录办理借出的馆员，也可能记录办理归还的馆员；可在借阅记录中设置 `borrowLibrarianNo` 和 `returnLibrarianNo`。
- 读者与图书之间通过“预约”联系形成 M:N。联系属性包括预约时间、预约状态。

关键约束：

- 一个馆藏副本同一时刻最多只能被一个读者借出，可通过副本状态和未归还借阅记录约束实现。
- 预约针对图书而不是副本，因为读者通常预约某本书，而不是指定某个副本。
- 归还馆员在尚未归还时可以为空。

### （2）关系模式转换

```text
Reader(readerNo, readerName, readerType, phone)
```

主码：`readerNo`。

```text
Book(bookNo, title, author, publisher, publishYear, classNo)
```

主码：`bookNo`。

```text
Copy(copyNo, bookNo, inDate, location, status)
```

主码：`copyNo`。外码：`bookNo` 参照 `Book(bookNo)`。

```text
Librarian(librarianNo, librarianName, position)
```

主码：`librarianNo`。

```text
Borrow(borrowNo, readerNo, copyNo, borrowTime, dueTime, returnTime, renewCount, borrowLibrarianNo, returnLibrarianNo)
```

主码：`borrowNo`。外码：`readerNo` 参照 `Reader(readerNo)`，`copyNo` 参照 `Copy(copyNo)`，`borrowLibrarianNo` 和 `returnLibrarianNo` 参照 `Librarian(librarianNo)`。

```text
Reservation(readerNo, bookNo, reserveTime, reserveStatus)
```

主码可取：`(readerNo, bookNo, reserveTime)`。外码：`readerNo` 参照 `Reader(readerNo)`，`bookNo` 参照 `Book(bookNo)`。

## 四、关系模式规范化题

已知：

```text
R(A, B, C, D, E, F, G)
F = { A -> B, BC -> D, D -> E, E -> F, CG -> A }
```

### （1）候选码

右部属性集合为：

```text
{A, B, D, E, F}
```

没有出现在任何函数依赖右部的属性为：

```text
{C, G}
```

所以任一候选码必须包含 `C` 和 `G`。

计算 `(CG)+`：

```text
CG
由 CG -> A 得 ACG
由 A -> B 得 ABCG
由 BC -> D 得 ABCDG
由 D -> E 得 ABCDEG
由 E -> F 得 ABCDEFG
```

因此：

```text
候选码为 CG
```

### （2）BCNF 分解

在原关系 R 中，候选码是 `CG`。

检查函数依赖：

- `A -> B` 中 A 不是超码，违反 BCNF。
- `BC -> D` 中 BC 不是超码，违反 BCNF。
- `D -> E` 中 D 不是超码，违反 BCNF。
- `E -> F` 中 E 不是超码，违反 BCNF。
- `CG -> A` 中 CG 是候选码，满足 BCNF。

按 `A -> B` 分解：

```text
R1(A, B)
R2(A, C, D, E, F, G)
```

`R1` 中 A 是码，满足 BCNF。

在 `R2` 中，`E -> F` 违反 BCNF，继续分解：

```text
R3(E, F)
R4(A, C, D, E, G)
```

`R3` 中 E 是码，满足 BCNF。

在 `R4` 中，`D -> E` 违反 BCNF，继续分解：

```text
R5(D, E)
R6(A, C, D, G)
```

`R5` 中 D 是码，满足 BCNF。

在 `R6` 中，投影得到的主要依赖包括：

```text
CG -> A
CG -> D
```

`CG` 是 `R6` 的码，所以 `R6` 满足 BCNF。

最终一组 BCNF 分解为：

```text
R1(A, B)
R3(E, F)
R5(D, E)
R6(A, C, D, G)
```

说明：BCNF 分解保证无损连接，但不一定保持全部函数依赖；这是 BCNF 相比 3NF 的常见代价。

## 五、可选考查题

### （1）关系代数优化

应尽早执行选择和投影，减少参与连接的数据量。

可先对三个关系分别筛选：

```text
Student' = σ_dNo='D01'(Student)
Course'  = σ_type='必修'(Course)
SC'      = σ_score>=80(SC)
```

再执行连接：

```text
π_sNo,sName,cName,score(Student' ⋈ SC' ⋈ Course')
```

这样可以减少连接前的元组数量和中间结果规模。

### （2）索引建议

可以考虑建立：

- `Student(dNo)`：用于快速筛选学院为 `D01` 的学生。
- `Student(sNo)`：通常主码已有索引，用于与 `SC` 连接。
- `SC(sNo)`：用于按学生连接选课记录。
- `SC(cNo)`：用于按课程连接课程信息。
- `SC(score)`：用于筛选成绩大于等于 80 的记录，若该条件选择率较高则收益有限。
- `Course(cNo)`：通常主码已有索引，用于连接。
- `Course(type)`：用于筛选必修课程。

如果只能建少量索引，优先考虑连接键和选择性较高的筛选条件，例如 `Student(dNo)`、`SC(sNo)`、`SC(cNo)`、`Course(cNo)`。

### （3）低基数属性索引

`Course.type` 只有少数固定取值，属于低基数属性，适合建立位图索引。位图索引适合取值种类少、读多写少、常用于组合筛选的属性。若系统更新频繁，普通 B+ 树索引或不建索引也可能更合适。
